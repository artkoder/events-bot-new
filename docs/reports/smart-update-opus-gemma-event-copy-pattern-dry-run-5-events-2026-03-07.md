## 2. Critical Verdict

Этот dry-run уже отвечает на главный вопрос: **новый подход теперь реально протестирован на реальных событиях**.

Но второй вывод не менее важен:

- **в код бота переносить его как есть нельзя**;
- prototype уже показывает, где pattern-driven branch сильнее baseline;
- и так же явно показывает, где он пока сырой или даже хуже baseline.

### Что выглядит promising

- `2687` — реальный quality/covarage win:
  - `missing: 5 -> 1`
  - сервисного leakage нет
  - текст стал плотнее по фактам
- `2673` — тоже сильный candidate:
  - `missing: 5 -> 1`
  - структура логично переключилась в `program_led`
  - но остался `cta_or_hashtag` leakage, значит branch ещё не production-ready
- `2734` — mixed improvement:
  - `missing: 5 -> 3`
  - исчез service-like leakage из baseline (`Возрастное ограничение`, `Продолжительность не уточняется`)
  - но extraction prototype оказался слишком агрессивным и срезал часть полезных фактов

### Что пока слабое

- `2660` — новый branch не дал quality win:
  - `missing: 2 -> 3`
  - routing ушёл в `value_led`, хотя кейс скорее просится в более аккуратный `topic_led`/`scene_led`
  - extraction/copy_assets тоже выглядят шумно (`format_signal = встреча` вместо выставки)
- `2745` — новый branch не помогает:
  - `missing: 5 -> 5`
  - текст короче и беднее baseline
  - бедный source не был компенсирован новым routing/prompting

### Главные проблемы prototype v1

- extraction prototype пока слишком агрессивно режет facts на части событий;
- `value_led` срабатывает слишком часто;
- copy_assets местами шумные или editorialized;
- revise/repair иногда лечит coverage ценой повторов и тяжёлого ритма;
- CTA hygiene ещё не закрыта до конца (`2673`);
- poor-source cases новая ветка пока не спасает.

### Вывод по имплементации

После этого dry-run мой verdict такой:

- **внедрять в бот весь prototype сейчас нельзя**;
- **внедрять ничего без такого dry-run действительно было бы ошибкой**;
- теперь у нас наконец есть основание для следующего шага:
  - либо сделать ещё один узкий раунд с Opus уже на этих реальных output-кейсах;
  - либо локально доработать prototype и прогнать второй comparison round.

## 4. Выводы

- Этот prototype уже решает главную проблему прошлого шага: теперь можно смотреть не только baseline current flow, а реальный output нового pattern-driven направления.
- Но это ещё не основание сразу встраивать всё в бот: сначала нужно редакторски сравнить тексты и решить, что именно из prototype действительно даёт quality lift.
- Если новый output выглядит сильнее, следующий шаг — переносить именно этот branch в код как экспериментальную реализацию, а не переписывать runtime вслепую.
- Если в prototype останутся coverage/grounding проблемы, их уже рационально отправлять в Opus вместе с этим отчётом, потому что здесь спор пойдёт по реальным результатам.
