# Fact-first Smart Update (внедрено)

Цель: перевести Smart Update на **строгий fact-first** пайплайн, где публичный текст (`description`, а также `short_description`/`search_digest`) строится **только** из извлечённых фактов, а не из “сырых” текстов источников.

Статус: **внедрено в runtime Smart Update** под флагом `SMART_UPDATE_FACT_FIRST` (по умолчанию включено).

---

## Определения и источники истины

**Источники фактов для события:**

- Все `event_source.source_text` (каждый источник: TG/VK/парсер/legacy‑снапшот и т.п.).
- Все OCR‑тексты афиш: `eventposter.ocr_text` (считать полноценным источником фактов).

Примечание про legacy события (до появления `event_source_fact`):
- Если у события есть только старое `description` (например импорт из `/parse`) и **нет** ни одного `event_source_fact`,
  Smart Update при первом merge сохраняет компактный снапшот в `event_source` с `source_type=legacy` и
  извлекает из этого legacy текста атомарные факты, записывая их в `event_source_fact`.
- Извлечение фактов из legacy-описания пропускается, если оно слишком короткое (защита от галлюцинаций): порог задаётся через
  `SMART_UPDATE_LEGACY_DESC_FACT_EXTRACT_MIN_CHARS` (default: `160`).
- Эти legacy‑факты используются в fact‑first генерации `description` (иначе при пере-сборке из фактов можно потерять детали,
  которые раньше жили только в `description`).
- При этом legacy‑источник **не** подмешивается в `facts_before` для LLM merge (чтобы снапшот не “заменял” реальные источники),
  но учитывается в fact‑first как baseline.

**Корзины фактов:**

- `facts_infoblock` — якоря/логистика и всё, что нельзя публиковать в нарративе:
  - дата/время/город/площадка/адрес,
  - билеты/вход/регистрация/запись (и ссылки),
  - точные цены/донаты,
  - возрастные ограничения,
  - «Пушкинская карта»,
  - контакты,
  - служебные факты про афиши (`Добавлена афиша: <url>`).
- `facts_text_clean` — **только публикуемые** содержательные факты для нарратива:
  - о чём событие, формат, участники,
  - программа/перечни (с сохранением элементов),
  - условия посещения/участия (длительность, что взять, что предоставляется, язык/субтитры, обсуждение после, правила перед началом, количество мест/участников/игроков),
  - допускается контентная ссылка на плейлист Я.Музыки, если она есть в фактах (только `music.yandex.ru/users/.../playlists/...`); остальные URL остаются запрещены,
  - чаты/сообщества участников можно упоминать как факт, но URL чата не публикуем в нарративе,
  - допускаются **умеренные оценочные** факты и **одна цитата**, если они явно в источнике и не являются CTA/гиперболой.
  - императив/CTA‑формулировки из источника (например «Создайте…», «Приходите…») должны либо попадать в `facts_drop`, либо конвертироваться в нейтральные факты вида «Можно…/Участники смогут…» без призыва.
- `facts_drop` — промо/CTA/хэштеги/шум/чужие события (особенно из постов‑расписаний).

**Главный инвариант:** `description_md` генерируется строго **только из `facts_text_clean`** (без подтягивания новых фактов из `source_text`).

## Пайплайн (runtime)

### 0) Переключатель (ENV)

- `SMART_UPDATE_FACT_FIRST=1` (default) — включён fact‑first для Smart Update (create + merge).
- `SMART_UPDATE_FACT_FIRST=0` — rollback на предыдущий rewrite/merge‑first путь (описание строится из сырых текстов источников).

Дополнительно:
- `SMART_UPDATE_DESCRIPTION_EMOJI_MODE=light|off` управляет лёгкими эмодзи в `description` (в `search_digest` они запрещены промптом).

### 1) Extract facts (LLM → JSON)

LLM (Gemma) извлекает атомарные факты из текста кандидата, включая OCR афиш. Факты используются:

- для лога источника (`event_source_fact`);
- как “канонический” набор для сборки нарратива.

Ключевое правило: **не включать** якорные поля (дата/время/город/площадка/адрес) в атомарные факты (они логируются и мерджатся детерминированно через поля `event.*` и отдельный инфоблок UI).

Примечание про списки: чтобы в fact‑first не терялись короткие программы/перечни, которые в источнике часто идут **не как буллеты**, а как “голые” строки (например список фильмов вида `«Название» — Автор`), в промпте извлечения фактов есть отдельное правило: такие пункты должны возвращаться как **отдельные facts** (verbatim, с сохранением порядка).

### 2) Bucket facts (guardrails, deterministic)

В runtime-коде факты **детерминированно** классифицируются в корзины:

- `facts_infoblock` — логистика/регистрация/ссылки/контакты/цены/возраст/«Пушкинская карта»/упоминания афиш;
- `facts_text_clean` — публикуемые нарративные факты;
- `facts_drop` — CTA/хэштеги/шум.

Дополнительно (важно для строгости):

- `facts_text_clean` очищается от якорей не только по префиксам `Дата:/Время:/Локация:/Адрес:/Город:`,
  но и по **форматам** дат/времени (например `2026-03-03`, `03.03`, `3 марта`, `18:30`) и по типовым маркерам
  места проведения (`зал`, `этаж`, `аудитория`, `по адресу`).
- На этапе формирования `facts_text_clean` применяется фильтр по **anchor‑строкам события**
  (дата/время/город/площадка/адрес): факты, содержащие эти якоря, не попадают в нарратив.

Корзины сейчас **не сохраняются отдельными полями** в БД: в `event_source_fact` хранится плоский лог, а `facts_text_clean` вычисляется на лету для генерации текста.

### 3) Generate `description` (LLM, строго из `facts_text_clean`)

`description` генерируется в стиле варианта **C+D** (из dry‑run):

- (опционально) эпиграф‑цитата в blockquote перед первым `###`;
- лид‑абзац (1–2 предложения) без заголовка (одним абзацем, без лишних переносов строки);
- 2–3 секции `### ...` с информативными подзаголовками (без общего «Подробности»);
- под каждым `###` — 2+ предложения или список (2+ пунктов); однотипные темы/пункты лучше группировать списком, а не дробить на микро‑разделы;
- 1–2 уместных эмодзи (без “ёлки”).

Жёсткие запреты в `facts_text_clean` и `description`:

- нельзя дата/время/город/площадка/адрес;
- нельзя URLs/телефоны (исключение: ссылка на плейлист Я.Музыки `music.yandex.ru/users/.../playlists/...` если она присутствует как факт);
- нельзя точные цены/донаты;
- нельзя “билеты/вход/регистрация/запись”;
- нельзя возрастные ограничения;
- нельзя “Пушкинская карта”;
- нельзя упоминать афиши;
- нельзя клише “это … не …, а …”.

### 4) Coverage-check + revise loop (LLM)

После генерации запускается LLM‑проверка покрытия (один раз):

- `missing[]` = факты из `facts_text_clean`, не отражённые в описании;
- `extra[]` = утверждения, которых нет в фактах (строго).

Затем (если нужно) выполняется 1 pass редактуры `description_md` с учётом `missing/extra` и детерминированных policy‑checks
(структура: лид + 2–3 `###`, отсутствие «Подробности», отсутствие микро‑секций, отсутствие forbidden‑markers).

Опционально допускается ещё один финальный pass, если после первой редактуры остаются проблемы со структурой/запретами
(например упрямые «посвящ…» или повторяющиеся заголовки).

Ожидаемый итог: `missing=[]` и `extra=[]`, плюс отсутствие forbidden‑markers (детерминированный чек).

Примечание: для устойчивости добавлен “last‑mile” шаг, который отдельно вычищает упрямые лексические утечки
вроде корня `посвящ…` (посвящён/посвящена/…), если модель продолжает использовать этот оборот после основных раундов.

### 5) Derived fields: `short_description` и `search_digest` (LLM)

Для удобства чтения и поиска:

- `short_description`: ровно 1 предложение на **12–16 слов**, без логистики и без эмодзи.
- `short_description`: не повторять заголовок события дословно, не использовать повелительное наклонение/CTA.
- `search_digest`: 1 предложение на **25–55 слов** (если нужно, допустимо **20–80**), без логистики, без эмодзи и без кавычек‑цитат.

Оба поля валидируются детерминированно (формат/длина + запреты).

## Где реализовано

Runtime:

- `smart_event_update.py`:
  - флаг `SMART_UPDATE_FACT_FIRST`;
  - извлечение фактов: `_llm_extract_candidate_facts(...)`;
  - генерация `description` строго из фактов (C+D): `_llm_fact_first_description_md(...)`.

---

## Примерная стоимость (LLM calls per event)

Зависит от количества источников/фактов и от того, проходят ли coverage/revise раунды с первого раза.

Оценка порядка величины:

- create: 1 запрос на facts‑JSON (extract) + 2–4 запроса на `description` (generate + coverage + 0–2 revise) + 2 запроса на `short_description`/`search_digest`.
- merge: 1 запрос на merge‑JSON + 2–4 запроса на `description` (generate + coverage + 0–2 revise) + 0–2 запроса на `short_description`/`search_digest` (если нужен refresh).
Фактический максимум на `description` держится в пределах 2–4 вызовов: это важно для лимитов Gemma и скорости VK/TG импорта.

## Где это сейчас проверяется (исследовательский прогон)

В репозитории есть исследовательский скрипт, который повторяет контракт пайплайна на фиксированном наборе событий и пишет отчёт:

- Пайплайн: `artifacts/codex/fact_first_gemma_dryrun_v3.py`
- Выход: `artifacts/codex/fact_first_gemma_5events_<DATE>_v*.md`

Отчёт содержит для каждого события:

- источники (`event_source`) + афиши/OCR,
- `facts_infoblock`, `facts_text_clean`, `facts_drop`, `conflicts`,
- `description_md`,
- `short_description`, `search_digest`,
- coverage‑итоги (`missing/extra/forbidden`).

---

## Миграция/rollback

- Чтобы быстро откатить поведение Smart Update на прежнюю схему (rewrite/merge‑first), выставьте `SMART_UPDATE_FACT_FIRST=0`.
- При возврате в fact‑first (`SMART_UPDATE_FACT_FIRST=1`) описание снова будет строиться строго из фактов.
