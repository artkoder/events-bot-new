# Claude CLI

Короткая шпаргалка по локальному `claude` CLI для консультаций, second opinion и одноразовых headless-запросов прямо из терминала.

Для двухэтапного workflow по trigger phrase `комплексная консультация` см. канонику `docs/tools/complex-consultation.md`.

В пользовательских запросах короткое слово `Opus` считается каноническим алиасом именно для strict-вызова `Claude CLI` с полным model id `claude-opus-4-6`.

## Когда использовать

- Нужна внешняя консультация по архитектуре, качеству текста, дебагу или trade-off.
- Нужен быстрый разовый запрос без веб-интерфейса.
- Нужен строгий review по brief, dry-run или набору гипотез.

## Проверка установки

Проверка бинарника и версии:

```bash
command -v claude
claude --version
```

Базовый help:

```bash
claude --help
```

## Базовые команды

Интерактивная сессия:

```bash
claude
```

Один запрос без интерактива:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Сделай краткий technical review этого плана"
```

Тот же смысл, если в разговоре пользователь пишет просто `спроси Opus` или `запусти Opus`.

JSON-ответ для проверки модели и учёта:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Какая модель сейчас отвечает? Ответь только точным идентификатором."
```

Запрос с контекстом из stdin:

```bash
cat artifacts/codex/tasks/brief.md | claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Сделай строгий review. Найди риски, регрессии и слабые предположения."
```

## Рекомендуемый режим для консультаций из репозитория

Для консультаций по проекту через `Opus` по умолчанию используй strict-режим:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json "..."
```

Почему именно так:

- `-p` даёт одноразовый headless-ответ;
- полный model id фиксирует именно `claude-opus-4-6`;
- `--tools ""` отключает built-in tools;
- `--output-format json` позволяет строго проверить `modelUsage`.

## Алиас в запросах

Правило маршрутизации для агента:

- `Opus` в пользовательском запросе означает strict-консультацию через `Claude CLI` с `--model claude-opus-4-6`
- агент не должен трактовать `Opus` как произвольную модель или выбирать другую Claude-модель вместо неё
- если пользователь просит `Opus`, безопасный дефолтный вызов это:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json "..."
```

## Strict Opus Only

Если нужна гарантия `только Opus`, используй более жёсткий режим:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json "..."
```

Почему так:

- полный model id убирает двусмысленность alias;
- `--tools ""` отключает built-in tools;
- не используется `--permission-mode plan`;
- не нужно задавать `--fallback-model`.

Проверка обязательна:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json "Пинг" \
  | jq -r '.modelUsage | keys[]'
```

Для `Opus`-консультации ожидаемый результат ровно один:

```text
claude-opus-4-6
```

Если в `modelUsage` появился любой другой ключ, такую консультацию нельзя считать `Opus-only`: её нужно перезапустить в strict-режиме и не опираться на старый ответ как на чистую консультацию `Opus`.

## Антидубль

Главный operational risk для `Opus` здесь не только в модели, но и в процессе запуска: если агент не дождался статуса первого запроса и тут же отправил fallback, токены тратятся дважды.

Правила:

- сначала собери один staged brief в отдельный файл;
- затем сделай один canonical launch в один целевой JSON-файл;
- до любого retry сначала проверь, завершён ли первый запуск, валиден ли JSON и что лежит в `modelUsage`;
- не запускай slim/fallback вариант параллельно или “на всякий случай”.

Плохой паттерн:

- отправить полный `Opus`
- не дождаться явного статуса
- сразу отправить второй, “более плотный” `Opus`

Правильный паттерн:

1. собрать brief
2. запустить один `Opus`
3. дождаться результата или явного failure state
4. только потом, если нужно, собирать retry-brief и делать второй запуск

Минимальная проверка перед retry:

```bash
ls -lh artifacts/codex/reports/<slug>-claude-opus.json
jq -r '.modelUsage | keys[]' artifacts/codex/reports/<slug>-claude-opus.json
jq -r '.result' artifacts/codex/reports/<slug>-claude-opus.json | sed -n '1,40p'
```

Если JSON уже полноценный и `modelUsage` чистый, второй запуск запрещён.

## Долгое ожидание нормально

Для `Opus` в strict `claude -p` режиме отсутствие промежуточного вывода не является признаком сбоя.

Нормально, если:

- CLI молчит до завершения;
- большой context pack считается долго;
- reasoning-модель на тяжёлом prompt review или архитектурном разборе думает заметно дольше обычного.

Поэтому:

- не запускай fallback только потому, что “слишком долго”;
- не трактуй тишину как failure state;
- заранее выбери timeout policy под размер evidence-пакета и придерживайся его;
- до истечения этого timeout policy не запускай второй `Opus`, если нет явного признака ошибки.

Хороший operational pattern:

1. выбрать timeout policy до запуска
2. стартовать один `Opus`
3. ждать до процесса/timeout/валидного JSON
4. только потом принимать решение о retry

Плохой pattern:

1. запустить `Opus`
2. увидеть тишину
3. решить, что он “завис”
4. отправить второй `Opus`

## Как передавать большой контекст

Для strict `Opus`-режима не полагайся на список путей или ссылок на файлы как на основной способ передачи контекста:

- с `--tools ""` модель не должна сама читать файлы;
- file refs полезны агенту и человеку, но не заменяют содержимое brief;
- нужные доказательства должны быть staged в одном self-contained brief.

Практический подход:

- сначала сделать curated brief;
- включать в него только релевантные разделы, а не все промежуточные версии подряд;
- держать full-fidelity evidence, но в компактной структуре: prompt context, grounded review, selected case evidence, ключевые симптомы, конкретные вопросы;
- заранее решить, какой timeout policy соответствует этому объёму контекста, и не ломать запуск посередине только из-за долгого thinking time;
- если контекст слишком тяжёлый, не отправлять “ещё один Opus сразу”, а сначала пересобрать brief плотнее и только потом делать retry.

## Когда просить prompt-review

Если задача реально про поведение LLM, деградацию текста, prompt quality или странные ответы модели, особенно `Gemma`, полезно добавить в запрос к `Opus` отдельную просьбу:

- оценить качество текущего prompt'а
- указать, какие формулировки создают нежелательное поведение
- предложить конкретные prompt-level правки, а не только общие советы

Это не нужно добавлять в каждый запрос. Делай так только там, где проблема plausibly лежит в prompt design или prompt/context assembly.

Пример:

```bash
cat artifacts/codex/tasks/brief.md | claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Сделай строгий инженерный review. Если видишь, что проблема может быть в prompt design или LLM framing, особенно для Gemma, отдельно оцени качество prompt'а и предложи конкретные правки prompt'а с объяснением, какие симптомы они должны исправить."
```

## Как проверить, какая модель реально ответила

Для `claude` удобнее всего смотреть JSON-результат: там есть поле `modelUsage` с точным resolved model id.

Пример:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Какая модель сейчас отвечает? Ответь только точным идентификатором." \
  | jq -r '.modelUsage | keys[]'
```

Это показывает фактическую модель, которая сгенерировала ответ, а не только текст саморассказа модели.

Если нужен полный JSON:

```bash
claude -p --model claude-opus-4-6 --tools "" --output-format json "Пинг"
```

Полезные поля:

- `result` — текст ответа;
- `session_id` — id сессии;
- `modelUsage` — фактическая модель и токены;
- `total_cost_usd` — стоимость запроса.

## Локальные настройки

Claude CLI хранит пользовательские настройки в:

```bash
~/.claude/settings.json
```

Быстрый просмотр:

```bash
cat ~/.claude/settings.json
```

Если нужно проверить текущий дефолтный alias модели, сначала смотри сюда, а затем подтверждай фактический resolved model через `--output-format json`.

## Практические замечания

- Для консультаций по этому репозиторию запускай `claude` из корня проекта.
- Для `Opus`-консультаций используй strict one-shot режим с `claude-opus-4-6` и `--tools ""`.
- Если нужен точный учёт модели и стоимости, добавляй `--output-format json`.
- Если нужно передать большой контекст, удобнее pipe через stdin.

## Артефакты

Если консультация влияет на решение по проекту:

- brief складывай в `artifacts/codex/tasks/`
- итоговый review/report складывай в `artifacts/codex/reports/`

Не коммить локальные пользовательские данные из `~/.claude/`.
