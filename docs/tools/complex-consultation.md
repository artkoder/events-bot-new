# Комплексная консультация

Канонический workflow для случаев, когда в запросе пользователя есть фраза `комплексная консультация`.

В этих инструкциях слово `Opus` используется как короткий алиас для strict-консультации через `Claude CLI` с полным model id `claude-opus-4-6`.

Для этапа `Opus` по умолчанию используй strict one-shot режим из `docs/tools/claude-cli.md`: полный model id `claude-opus-4-6`, `--tools ""`, без `plan`, и с проверкой `modelUsage`.

Под этим понимается конкурентная внешняя консультация в два этапа:

1. сначала `Opus` (`Claude CLI` с `--model claude-opus-4-6`)
2. затем `Gemini` с `gemini-3.1-pro-preview`, которому передаются и материалы задачи, и отчёт `Opus`

Цель: получить два сильных инженерных мнения с разной оптикой, а затем самостоятельно проверить их выводы по коду, тестам, логам и документации.

## Когда запускать

Запускай этот цикл по умолчанию, если пользователь прямо пишет `комплексная консультация` и не задаёт другой порядок или состав моделей.

Если пользователь в формулировке пишет `Opus`, трактуй это как первый этап через strict-вызов `Claude CLI` с `--model claude-opus-4-6`.

Если пользователь явно ограничил консультацию одной моделью, следуй его указанию.

## Принципы

- Это не запрос на подтверждение текущей гипотезы.
- Внешние модели должны иметь право спорить, возражать и предлагать более сильные варианты.
- Их нужно просить давать инженерный, критичный и practically useful разбор.
- Их выводы нельзя принимать на веру без собственной проверки.
- Итоговое решение принимает агент после сверки с репозиторием и фактами.
- Если задача materially связана с prompt behavior или качеством LLM-текста, особенно у `Gemma`, в этапе `Opus` уместно добавить отдельную просьбу про review качества prompt'ов и конкретные prompt-level правки.
- Но это не mandatory для каждой консультации: prompt-audit добавляется только там, где он реально релевантен задаче.

## Фиксированный порядок

1. Подготовь самодостаточный brief с задачей, целями, ограничениями, текущей гипотезой, фактами, артефактами и вопросами.
2. Отправь этот brief в `Opus`.
3. Сохрани полный отчёт `Opus`.
4. Отправь в `Gemini 3.1 Pro` тот же brief, плюс краткий контекст от агента, плюс полный отчёт `Opus`.
5. Сопоставь ответы, проверь спорные места по коду и данным, подготовь итоговый synthesis.

## Что должно быть в brief

- задача и желаемый результат
- текущая реализация или текущая гипотеза
- ограничения и инварианты
- симптомы, регрессии, факты, логи, dry-run evidence
- релевантные file refs
- что именно нужно критиковать или улучшать

Важно:

- brief должен быть self-contained для strict `Opus`-режима;
- не рассчитывай, что `Opus` сама откроет файлы по списку путей при `--tools ""`;
- file refs можно оставлять как provenance, но ключевое содержимое должно быть уже staged внутри brief.

Рекомендуемое место:

```bash
artifacts/codex/tasks/<slug>-complex-consultation-brief.md
```

## Антидубль для Opus

Перед этапом `Opus` агент обязан:

1. собрать один canonical brief
2. выбрать один canonical output path
3. выполнить один запуск
4. проверить статус этого запуска
5. только после этого решать, нужен ли retry

Нельзя:

- запускать второй `Opus`, пока первый ещё может быть жив;
- запускать slim/fallback вариант “на всякий случай” до проверки первого JSON;
- смешивать в одном цикле несколько одновременных `Opus`-прогонов ради подстраховки.

Retry допустим только если первый запуск:

- явно упал;
- дал битый/пустой JSON;
- завершился без usable `result`;
- вышел за заранее выбранный timeout и это подтверждено, а не предположено.

Если первый запуск уже дал валидный JSON с чистым `modelUsage=claude-opus-4-6`, второй запуск запрещён.

Важно:

- тишина в `claude -p` не считается failure signal;
- долгий runtime для `Opus` на тяжёлом evidence-пакете сам по себе нормален;
- timeout policy должен быть выбран до запуска, а не “по ощущениям в процессе”.

## Рамка для Opus

`Opus` идёт первым и получает только материалы задачи и контекст от агента.

Для этого этапа не используй `plan` или tool-enabled сценарии.

Рекомендуемая рамка:

- ты внешний инженерный консультант
- не соглашайся автоматически с текущей гипотезой
- если видишь лучший путь, предложи его
- укажи риски, регрессии, слабые допущения и missing evidence
- дай подробный отчёт, а не короткий verdict
- если задача связана с LLM behavior или prompt quality, особенно для Gemma, отдельно оцени качество prompt'ов и предложи конкретные доработки

Рекомендуемая команда:

```bash
cat artifacts/codex/tasks/<slug>-complex-consultation-brief.md | \
  claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "Ты внешний инженерный консультант. Не соглашайся автоматически с моей гипотезой. Спорь, если видишь слабые места. Предлагай более сильные решения, если они лучше ведут к цели. Верни подробный отчёт с разделами: diagnosis, risks, disagreements, alternative proposals, recommended next steps, open questions."
```

Операционно безопаснее сначала собрать brief в файл, а затем отправлять именно этот файл в одном запуске. Не делай “full brief” и “slim brief” параллельно.

Если `Opus` долго думает, это не повод автоматически запускать slim/fallback вариант. Сначала дождись либо валидного результата, либо заранее определённого failure state.

Если задача prompt-centric, расширяй запрос к `Opus` дополнительным требованием:

```text
Если видишь, что проблема plausibly связана с prompt design, prompt framing или prompt/context assembly, особенно для Gemma, отдельно сделай review качества prompt'а и предложи конкретные prompt-level изменения с объяснением ожидаемого эффекта.
```

Сохранение результата:

```bash
cat artifacts/codex/tasks/<slug>-complex-consultation-brief.md | \
  claude -p --model claude-opus-4-6 --tools "" --output-format json \
  "..." \
  > artifacts/codex/reports/<slug>-claude-opus.json

jq -r '.result' artifacts/codex/reports/<slug>-claude-opus.json \
  > artifacts/codex/reports/<slug>-claude-opus.md
```

Проверка фактической модели:

```bash
jq -r '.modelUsage | keys[]' artifacts/codex/reports/<slug>-claude-opus.json
```

Здесь должен быть ровно один ключ: `claude-opus-4-6`.

Перед любым retry сначала проверь:

```bash
ls -lh artifacts/codex/reports/<slug>-claude-opus.json
jq -r '.modelUsage | keys[]' artifacts/codex/reports/<slug>-claude-opus.json
jq -r '.result' artifacts/codex/reports/<slug>-claude-opus.json | sed -n '1,40p'
```

Если JSON уже usable, следующий `Opus` запуск не нужен.

## Рамка для Gemini 3.1

Gemini идёт второй и получает:

- исходный brief
- краткий контекст от агента
- полный отчёт `Opus`

Gemini не должна просто повторять `Opus`. Её задача:

- критично проверить выводы `Opus`
- спорить там, где аргументация слабая
- добавлять альтернативы и дополнительные углы зрения
- указывать, где нужно больше evidence

Рекомендуемая команда:

```bash
{
  cat artifacts/codex/tasks/<slug>-complex-consultation-brief.md
  printf '\n\n## Контекст от агента\n\n'
  printf '%s\n' '<короткое резюме того, что уже понял агент и где есть сомнения>'
  printf '\n\n## Отчёт Opus\n\n'
  cat artifacts/codex/reports/<slug>-claude-opus.md
} | gemini -m gemini-3.1-pro-preview -p "
Ты внешний инженерный консультант.
Тебе даны материалы задачи и полный отчёт Opus.
Не соглашайся автоматически ни со мной, ни с Opus.
Спорь, если видишь слабые места или лучший путь.
Предлагай свои решения, если они лучше ведут к цели.
Верни подробный отчёт с разделами: diagnosis, critique of Opus, disagreements, alternative proposals, recommended next steps, open questions.
"
```

Рекомендуемое место результата:

```bash
artifacts/codex/reports/<slug>-gemini-3.1-pro.md
```

Проверка фактической модели Gemini: см. канонику в `docs/tools/gemini-cli.md`.

## Что агент делает после двух консультаций

Агент обязан:

- сравнить выводы `Opus` и `Gemini`
- явно выделить совпадения и расхождения
- проверить спорные тезисы по коду, тестам, логам, dry-run и документации
- отделить подтверждённые факты от внешних гипотез
- предложить итоговое решение с собственной аргументацией

Нельзя:

- слепо принять любой внешний совет
- выдать внешнее мнение за проверенный факт
- заставлять внешние модели только подтверждать уже выбранный план

## Рекомендуемые артефакты

- brief: `artifacts/codex/tasks/<slug>-complex-consultation-brief.md`
- Opus raw JSON: `artifacts/codex/reports/<slug>-claude-opus.json`
- Opus report: `artifacts/codex/reports/<slug>-claude-opus.md`
- Gemini report: `artifacts/codex/reports/<slug>-gemini-3.1-pro.md`
- итоговый synthesis при необходимости: `artifacts/codex/reports/<slug>-consultation-synthesis.md`

## Короткая памятка

- trigger phrase: `комплексная консультация`
- order: `Opus` -> `Gemini 3.1 Pro`
- Gemini получает и материалы задачи, и отчёт `Opus`
- оба дают подробный инженерный отчёт
- агент верифицирует всё сам и не верит на слово
