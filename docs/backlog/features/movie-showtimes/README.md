# Movie Showtimes (фильмы и наличие в кинотеатрах)

> **Status:** Not implemented / In design  
> Цель: получать **метаданные фильма** и **расписание сеансов в кинотеатрах** из нескольких API, затем использовать это в боте (в т.ч. через Smart Update и дайджесты).

## Зачем это нужно

- Делать карточки кинопоказов богаче: название, постер, описание, возрастной рейтинг, длительность, жанры.
- Быстро отвечать на вопрос “где идёт фильм” (по городу/дате).
- Автоматизировать создание событий типа `кинопоказ` (см. `docs/architecture/overview.md`) из источников расписаний.

## Аналогия с фестивальной очередью

Как и фестивальная очередь (`docs/features/festivals/README.md`), эта фича предполагает отдельный “поток задач”:

- первичный сигнал (текст/ссылка/поиск по названию) создаёт задачу на обогащение;
- затем выполняются API-запросы, нормализация, матчинг и (в перспективе) импорт в события через Smart Update.

## High-level дизайн

### 1) Провайдеры (несколько API)

Разделяем источники на два типа:

- **Metadata provider**: отдаёт информацию о фильме (названия, описание, постеры, страны/жанры, возраст, длительность, участники и т.д.).
- **Showtimes provider**: отдаёт расписание/наличие в кинотеатрах (город, кинотеатр, время, цена/ссылка, форматы).

Дальше в коде это удобно оформлять как “плагины” (адаптеры) с единым нормализованным выходом.

### 2) Нормализованные сущности (идея модели)

- `Film` (каноническая сущность):
  - `title_ru/title_original`, `year`, `countries[]`, `genres[]`
  - `description`, `age_rating`, `runtime_minutes`
  - `poster_url`, `cover_url`
  - `external_ids` (например: `kinopoiskId`, `imdbId`, и т.п.)
- `Cinema`:
  - `name`, `city`, `address?`, `geo?`, `external_ids`
- `Showtime`:
  - `film_id`, `cinema_id`, `starts_at`, `language?`, `format?` (2D/3D/IMAX), `price?`, `ticket_url?`

Плюс обязательная таблица связей “провайдер → внешний id” (например `provider_film_ref`), чтобы не полагаться на fuzzy-match.

### 3) Очередь задач (как у фестивалей)

Минимальный набор задач (пока проектно):

1. `resolve_film`: вход: строка/ссылка/внешний id; выход: канонический `Film` + список `provider_film_ref`.
2. `fetch_showtimes`: вход: `Film` + город/дата; выход: `Showtime[]`.
3. `sync_events_via_smart_update` (опционально): создать/обновить события типа `кинопоказ` (через Smart Update, см. `docs/features/smart-event-update/README.md`).

## Kinopoisk Unofficial API (внесено сейчас)

- Провайдер: metadata (частично: премьеры и изображения; showtimes API в OpenAPI не обнаружен).
- Док: `docs/backlog/features/movie-showtimes/providers/kinopoisk-unofficial.md`

## Решения, которые нужно принять перед имплементацией

- Какие showtimes-провайдеры подключаем (1–2 на старт), и как легально/стабильно получать расписание по Калининграду.
- Где хранить кэш (SQLite vs Supabase) и какие TTL по сущностям (metadata обычно меняется редко, showtimes часто).
- Как маппить showtimes в события:
  - отдельное событие на каждый сеанс, или “событие-фильм” с расписанием внутри описания/кнопок.
