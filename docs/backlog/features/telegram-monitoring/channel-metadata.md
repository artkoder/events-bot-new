# Telegram Monitoring — метаданные источников (title/about/links) и автоподсказки

Статус: реализовано.
Каноническое описание текущего поведения: `docs/features/telegram-monitoring/README.md`.

Цель: улучшить флоу настройки фестивальных каналов (и каналов-организаторов), когда канал посвящён одному фестивалю и в **названии/описании** есть полезная информация (название серии, официальный сайт, соцсети, даты).

## Ограничение (важно)

Сервер бота **не должен** ходить в Telegram API за метаданными (title/about/links).
Единственное место, где эти данные доступны — **Kaggle TelegramMonitor** (Telethon), т.е. метаданные должны:

1) быть получены **во время запуска мониторинга** в Kaggle;
2) быть положены в `telegram_results.json`;
3) быть обработаны на сервере после скачивания результатов.

Следствие: при добавлении источника в `/tg` метаданных может не быть сразу; они появятся **после ближайшего запуска мониторинга**.

## Что считаем “метаданными источника”

- `title` — название канала/группы.
- `about` — описание (bio) канала/группы (full info).
- `links` — внешние ссылки, извлечённые из `about` (и опционально из закрепа).

## Автоподсказки (из метаданных)

Нужно извлечь и показать оператору (без автоперезаписи ручных настроек):

- `festival_series_suggestion` — короткое название **серии** (например: «Кантата», «Открытое море», «Гаражка»).
- `website_url_suggestion` — официальный сайт фестиваля (если в `about` есть ссылка).
- `aliases[]` — полезные алиасы серии (например “GARAZHKA Kaliningrad” → “Гаражка”).
- `confidence` + `rationale_short` — объяснение, почему предложено (1 строка).

Важно: подсказки **не должны** автоматически менять `telegram_source.festival_series`, если оператор уже выставил значение вручную.

## Изменение контракта `telegram_results.json` (план)

Добавить раздел с метаданными на уровень всего прогона, чтобы не дублировать одно и то же в каждом сообщении:

```json
{
  "schema_version": 2,
  "run_id": "kaggle_xxx",
  "generated_at": "2026-02-17T12:00:00Z",
  "sources_meta": [
    {
      "username": "open_fest",
      "source_type": "channel|group|supergroup|unknown",
      "title": "Open Fest …",
      "about": "…",
      "about_links": ["https://example.org", "https://vk.com/..."],
      "fetched_at": "2026-02-17T11:59:00Z",
      "meta_hash": "sha256:…",
      "suggestions": {
        "festival_series": "Открытое море",
        "website_url": "https://…",
        "aliases": ["Open Fest", "OpenFest"],
        "confidence": 0.86,
        "rationale_short": "Название канала и ссылка на сайт фестиваля в описании."
      }
    }
  ],
  "messages": [ /* как и раньше */ ],
  "stats": { /* как и раньше */ }
}
```

Правила совместимости:

- `schema_version=1` остаётся валидным (сервер должен принимать обе версии).
- `sources_meta` опционален (может отсутствовать).
- `message.source_title` остаётся (как сегодня), но `sources_meta[].title` становится каноничным источником метаданных.

## Что делает Kaggle (план)

- Для каждого `username` получить `title` (из entity) и `about` (из full info).
- Вычислить `meta_hash` по нормализованным `title+about`, чтобы понимать “изменилось ли”.
- Извлечь `about_links` детерминированно (regex).
- Прогнать LLM **один раз на источник** (не на сообщение) и получить `suggestions`.
- Сложить результат в `sources_meta[]` и записать в `telegram_results.json`.

Оптимизация/лимиты:

- Не дергать full info чаще, чем нужно: если мониторинг видит, что `title`/entity не менялся, можно пропускать `about` (best-effort).
- Для источников с `festival_series` уже заданным вручную можно снижать частоту LLM (но всё равно сохранять сырые `title/about/links`).

## Что делает сервер (план)

- При обработке `telegram_results.json`:
  - обновлять `telegram_source.title` **если изменился** (не только если он пустой);
  - сохранять `about`, `about_links`, `fetched_at`, `meta_hash` (требуются новые поля/таблица);
  - сохранять `suggestions.*` в отдельные поля (чтобы показать в UI) без перезаписи ручных настроек.
- В UI `/tg`:
  - показывать подсказку рядом с `🎪 Фестиваль` (и кнопку “Принять”);
  - показывать “Сайт” (если найден) как кнопку/ссылку.
